S2 Debounce + RAF no redesenho ✅ Feito (mar/2026, v27) 🟢 Simples ⚠️ Atenção ›

RAF gate implementado: draw() é limitado a 1 por frame (max 60fps). Inputs de texto usam drawDebounced() com 150ms de espera. Função original renomeada para _drawNow().

  • draw() usa requestAnimationFrame com flag _rafId — colapsa chamadas redundantes
  • drawDebounced() com 150ms para inputs de texto — evita redesenho a cada tecla
  • Drag da foto continua usando draw() direto (resposta imediata)
S3 Reduzir foto ao carregar ✅ Feito (mar/2026, v27) 🟢 Simples ⚠️ Atenção ›

Fotos maiores que 2160px são reduzidas via _downscaleFoto() antes de guardar em memória. Imagens com canal alpha (cutouts) são preservadas na resolução original.

  • Limite: 2160px no maior eixo — equivale a 2× canvas de 1080px
  • Skip automático se foto.isTransparent = true (detectado via detectaAlpha())
  • Downscale via canvas offscreen + toBlob(jpeg, 0.95) → novo Image

Bugs & Robustez N1–N5

Problemas encontrados na análise do v23 — não cobertos pelo backlog anterior.

N1 Troca rápida de layouts pode misturar elementos no canvas ✅ Feito (mar/2026, v27) 🟡 Médio ⚠️ Atenção ›

Token de sessão _layoutLoadToken implementado. Cada selectLayout() incrementa o token; após cada await, verifica se o token ainda é válido — se outro layout foi selecionado enquanto esperava, aborta silenciosamente.

  • const myToken = ++_layoutLoadToken no início de cada selectLayout()
  • Guards após await extractLayoutMeta() e await loadLayoutAssets()
  • Apenas o último layout clicado chega ao canvas
N2 Imagem travada pode deixar o canvas em branco para sempre ✅ Feito (mar/2026, v25) 🟢 Simples ✅ Seguro ›

Timeout de 15s adicionado em loadImg. Se a imagem não carregar no prazo, a promessa rejeita com 'timeout:url', o carregamento continua normalmente para as demais camadas.

  • setTimeout 15000 em loadImg — limpa o timer em onload/onerror
N3 Log interno acumula sem limite e deixa o programa mais lento com o tempo ✅ Feito (mar/2026, v25) 🟢 Simples ✅ Seguro ›

Painel visual de debug removido da tela (era exibido no canto inferior direito). Log continua sendo enviado ao servidor via POST — leitura por curl http://localhost:3333/log. Sem acúmulo na UI.

  • Elemento #_dbg removido de dbg() — só o envio remoto permanece
N4 Efeito glass recria elementos internos do zero a cada digitação ✅ Feito (mar/2026, v27) 🟡 Médio ⚠️ Atenção ›

Os três canvas offscreen do glass (_glSnap, _glBlurred, _glOff) são criados uma vez como variáveis de módulo e reutilizados a cada draw(). Redimensionados apenas quando W/H muda.

  • const _glSnap, _glBlurred, _glOff declarados junto aos globals
  • Redimensionamento condicional: if (canvas.width !== W) canvas.width = W
  • clearRect() antes de redesenhar — evita artefatos de frame anterior
N5 Muitos layouts podem travar o carregamento das miniaturas ✅ Feito (mar/2026, v27) 🟢 Simples ✅ Seguro ›

fetchThumbnails divide os layouts em lotes de 25, busca em paralelo via Promise.all e mescla os mapas de imagens antes de preencher os cards.

  • Chunking: for (let i = 0; i < layouts.length; i += 25)
  • Paralelo: Promise.all(chunks.map(async chunk => figmaGet(...)))
  • Merge: Object.assign({}, ...results) — seguro mesmo com chunks vazios
N6 Drag da foto travado no eixo horizontal ✅ Corrigido (mar/2026, v25) 🟢 Simples ⚠️ Atenção ›

Ao arrastar a foto no modo drag, a imagem só se movia horizontalmente. O eixo Y ficava travado.

  • Causa: bloco "auto-shift" em drawFoto clampava dy a cada render, cancelando o movimento vertical do usuário
  • Correção: auto-shift removido de drawFoto — posicionamento inicial corrigido via _aplicarAlinhamento (eye-aware) e defaultFotoOffset
N7 Foto não reposicionava ao trocar de layout ✅ Corrigido (mar/2026, v25) 🟢 Simples ✅ Seguro ›

Se uma foto já estava carregada e o usuário trocava de layout, a imagem ficava fora de posição — o alinhamento automático por olhos não era acionado na troca.

  • Ao finalizar loadLayerImages, se há foto carregada e o novo layout tem field:olhos, detectarAlinharOlhos() é chamado automaticamente
  • Cobre também o caminho onde não há camadas a carregar (retorno antecipado)
N8 Alinhar olhos não funcionava com foto espelhada (flipH) ✅ Corrigido (mar/2026, v27) 🟢 Simples ⚠️ Atenção ›

Com o flip horizontal ativado, o botão "Alinhar olhos" posicionava a foto no lugar errado — ou selecionava o rosto errado em fotos com múltiplas pessoas.

  • Causa: detecção rodava na imagem não-espelhada, mas _selecionarRosto() comparava coords naturais com olhosBox.cx (espaço de exibição) — lados trocados
  • Fix: detectarAlinharOlhos() desenha a imagem espelhada no canvas de detecção quando foto.flipH = true — coordenadas de saída já ficam no espaço de exibição
  • Removido natW - refPoint.x de _aplicarAlinhamento — não mais necessário
Performance P1–P3

Otimizações de carregamento e memória — implementadas em v27.

P1 Cache de imagens em memória ✅ Feito (mar/2026, v27) 🟢 Simples ✅ Seguro ›

Ao trocar de layout e voltar, todas as imagens das camadas eram baixadas novamente do Figma — mesmo sem nada ter mudado.

  • const _imgCache = new Map() global — url → HTMLImageElement
  • loadImg() verifica o cache antes de fazer fetch; armazena após carregar
  • Cache limpo em resetFigmaToken() para garantir dados frescos ao trocar de conta
  • Segunda visita a qualquer layout: camadas carregam instantaneamente da memória
P2 Assets do layout carregam em paralelo ✅ Feito (mar/2026, v27) 🟢 Simples ✅ Seguro ›

loadLayoutAssets() carregava os assets de um layout (logos, fundos, ícones) em sequência — um de cada vez. Com 3 assets, esperava 3× mais do necessário.

  • Loop for await substituído por Promise.all(slots.map(async ...))
  • Todos os assets do layout baixam em paralelo — reduz tempo proporcional ao número de assets
P3 Cache do mapa de URLs de camadas ✅ Feito (mar/2026, v27) 🟢 Simples ✅ Seguro ›

A chamada GET /images/?ids=...&scale=1 ao Figma retorna um mapa {nodeId: url} para todas as camadas. Era refeita a cada visita ao mesmo layout.

  • const _layerUrlCache = new Map() global — layoutId → mapa de URLs
  • loadLayerImages() reutiliza o mapa se já existe para o layout atual
  • Cache limpo junto com _imgCache ao resetar o token Figma
  • Com P1+P3 juntos: segunda visita ao mesmo layout é quase instantânea

Fix B Ícone da tag desalinhado na pill dinâmica ✅ Corrigido (mar/2026, v27) 🟡 Médio ⚠️ Atenção ›

O ícone asset/tag-icon dentro do blocoTag aparecia desalinhado quando o computeDynLayout reposicionava a pill verticalmente (pillDY ≠ 0). O texto da tag também ficava sem padding adequado à direita.

  • Detecção geométrica: layers dentro dos bounds de tagPillFrame (±20px) são puladas em drawWithLayerStack e redesenhadas com pillDY
  • icSlot: loadedAssets['tag-icon'] é desenhado em posição relativa ao pill — pf.x + icRelX, pf.y + pillDY + icRelY
  • Texto centralizado: textBaseline='middle' ancorado ao centro vertical do pill (pf.y + pillDY + pf.h/2) — mesma âncora do ícone
  • Texto X seguro: Math.max(tp.x, pf.x + icRelX + icSlot.w + 8) — nunca sobrepõe o ícone
  • padRight do Figma: (pf.x + pf.w) − (tp.x + tp.w) preserva o respiro original do designer
  • Escopo corrigido: tp, pillDY, icSlot todos declarados antes do uso

U2 Posicionamento automático por olhos ✅ Funcional (mar/2026) 🔴 Complexo ⚠️ Atenção ⏱️ ~1,5 dia ›

A ferramenta detecta o rosto na foto e posiciona automaticamente para os olhos caírem no ponto marcado pelo designer no Figma (field:olhos). A imagem nunca é cortada — sempre aspect-fit — e é escalada automaticamente para preencher a área visível sem espaços brancos.

O que foi implementado

  • Layer field:olhos no Figma define cx/cy (ponto-alvo) e w (IOD de referência)
  • face-api.js carregado via CDN GitHub (~1.5MB, browser cacheia; CDN npm não tem pasta weights/)
  • Detecção automática ao carregar foto + botão manual 🎯 Alinhar olhos
  • fillScale eye-aware: escala mínima calculada pela posição dos olhos na imagem para garantir que nenhuma borda da área visível fique com espaço branco
  • drawFoto sempre aspect-fit (crop removido) com clip ao fotoField
  • Mira azul (crosshair) confirma visualmente o ponto-alvo após alinhamento
  • Log remoto via dbg() → curl http://localhost:3333/log

Cadeia de fallbacks — 5 níveis

  • L1 — Dois olhos (rosto frontal): midpoint entre os olhos · Confiança: ALTA
  • L2 — Um olho (perfil): olho visível, escala pelo tamanho do rosto · Confiança: MÉDIA
  • L3 — Rosto sem olhos visíveis: centro do rosto (óculos escuros, baixa qualidade) · Confiança: BAIXA
  • L4 — Retry automático: reprocessa com contraste aumentado se L1-L3 falharam · Confiança: MÉDIA/BAIXA
  • L5 — Falhou: mantém posição atual, avisa o operador

U5 Geração em escala com IA (batch) 🔴 Complexo ⚠️ Atenção ⏱️ ~4 dias ›

Hoje a ferramenta é um editor manual: um banner de cada vez. Para campanhas com 10 peças × 3 formatos = 30 banners, você faz tudo na mão.

  • Módulo A — Batch: CSV/JSON com titulo, subtítulo, tag, cta, foto, layout → ZIP com tudo
  • Módulo B — Claude gera texto: você descreve o contexto → Claude retorna N variações em JSON → alimenta o Módulo A
  • Módulo C — Seleção de layout: Claude sugere o layout ideal pelo tipo/tom do texto

✅ Claude consegue: ler todos os frames do Figma, extrair tokens de design, entender brand guide, gerar copy, selecionar layout por objetivo.
❌ Claude não faz: criar imagens/fotos, criar vectors, avaliar "ficou bonito".

U6 Gerador + Publicador de Posts no Instagram 🔴 Complexo ⚠️ Atenção ⏱️ ~1 semana ›

Pipeline completo: usuário dá um briefing → Claude gera o copy → ferramenta escolhe o template e renderiza o banner → Instagram Graph API posta ou agenda. Evolução natural do U5.

  • Módulo A — Render headless: Puppeteer abre o gerador sem UI, preenche os campos via JS, exporta canvas.toDataURL() como PNG — sem reescrever o motor de render
  • Módulo B — Claude gera copy: briefing do usuário → Claude API retorna variações de título/subtítulo/tag/CTA em JSON
  • Módulo C — Seleção de template: Claude ou regra escolhe o layout ideal por cliente, formato e tom
  • Módulo D — Publicação: Instagram Graph API posta ou agenda — suporta agendamento nativo, requer conta Business/Creator + app no Meta Developer

Dependência: U5 é o predecessor — U6 é U5 + publicação automática.
✅ Não precisa reescrever o motor de render — Puppeteer reutiliza tudo que já existe.

U8 Auto-detecção de elementos Figma sem nomenclaturas obrigatórias 🔴 Complexo ⚠️ Atenção ⏱️ ~1 dia ›

Hoje o designer precisa renomear layers no Figma com nomes exatos (field:titulo, blocoGlass, asset/logo…) para que a ferramenta funcione. A ideia é inverter: a ferramenta analisa propriedades do nó (tipo, fills, effects, opacidade, hierarquia) e detecta automaticamente o papel de cada layer.

  • Alta confiança (90–100%): BACKGROUND_BLUR → blocoGlass · fill IMAGE → foto · opacity=0 TEXT → ctatx · opacity=0 não-TEXT → olhos
  • Heurística (70–80%): maior fontSize no container → título · segundo maior → texto · TEXT curto próximo a pill → tag
  • Múltiplos campos de texto (tx1, tx2, tx3…): peças específicas podem ter 3, 4 ou mais textos editáveis além do título e subtítulo — o sistema tx1/tx2/… já existe no código (buildFieldUI gera inputs dinamicamente, drawEditableText renderiza em loop). A auto-detecção precisa identificar e preencher esses campos extras em ordem de posição Y (de cima para baixo), após detectar título e texto
  • Abordagem híbrida: nome explícito sempre tem prioridade — heurística é fallback apenas para layers sem nome
  • Backward-compatible: layouts já nomeados continuam funcionando sem nenhuma mudança
  • Log indica quais campos foram detectados por nome vs. por heurística
⚠️ Casos ambíguos ainda precisarão de nome explícito — ex: dois textos do mesmo tamanho, dois containers grandes. Heurística não elimina a necessidade de nomes, apenas os torna opcionais para layouts simples.

M1 Mobile friendly — layout responsivo + touch 🔴 Complexo ⚠️ Atenção ⏱️ ~8h ›

A ferramenta foi pensada para desktop. Em celular, a sidebar lateral não cabe na tela, os botões são pequenos demais para toque e o drag da foto não responde ao dedo.

  • Layout: @media (max-width: 768px) empilha sidebar + canvas verticalmente — canvas no topo, controles abaixo
  • Canvas: zoom automático por CSS transform: scale para caber no viewport (similar ao S11)
  • Drag da foto: adicionar touchstart/touchmove/touchend ao lado dos mouse events existentes; pinch-to-zoom para o slider de escala
  • Galeria de fotos: grid responsivo com cards maiores para toque
  • Botões da toolbar: área de toque mínima de 44×44px (guideline iOS/Android)
  • Upload de foto: testar accept="image/*;capture=camera" para acesso direto à câmera no iOS e Android
  • Modais: centralizar e limitar largura a 95vw em telas estreitas
⚠️ Não pode quebrar a experiência desktop — toda mudança deve ser protegida por media query ou detecção de touch. Recomendado: branch separado para testes antes de merge.
Limitação conhecida — elementos vetorizados com blur (swoosh, decorativos)
Canvas 2D não reproduz filtros vetoriais do Figma (blur direcional, gradientes complexos, blend modes aninhados). Não é limitação desta ferramenta — é do canvas HTML.

✅ Solução atual: designer exporta como PNG com blur "assado" via asset/swoosh.
💡 Variação por tema: asset/swoosh-dark / asset/swoosh-light com toggle.
🔬 Alternativa futura: SVG+filter no DOM (mais complexo, possível para casos simples).
